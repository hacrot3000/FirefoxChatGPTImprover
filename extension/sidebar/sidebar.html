Automation rules

Rule runtime Not running
?
Each rule has its own monitor, target, baseline, cycle, pipeline, and optional command action. Automatic commands resolve an enabled preset in the background and never accept command text from the web page. A preset used automatically must have confirmation disabled.

Monitor element

?

The activation state is used only as a baseline and never triggers MATCHED by itself. Hidden includes display:none, visibility:hidden/collapse, the hidden, visible=false, aria-hidden=true attribute, or no rendered box.

You can remove or disable every attribute condition to track only the visibility transition.

The stability windows ignore brief state flicker caused by React re-rendering. A value of 0 keeps immediate behavior.

Use “Pick on page” to hover an element and click it to fill the selector; press Esc to cancel. The test button outlines selector matches in orange and condition matches in green.

New target element

?
Use “Pick on page” to select a target. On activation, or when the monitor leaves MATCHED, existing targets become the baseline and are not clicked. While MATCHED, only targets newer than the baseline are handled. Dry run still marks targets as handled for the cycle. Pending pipelines are cancelled when the tab is paused or stopped, the configuration changes, or the monitor re-arms.

Managed download

?

These settings belong to the selected local-action profile or the current tab override, not to the automation configuration profile.

Each target-triggered download freezes its destination and shell settings at click time. Later URL, profile, or tab edits cannot redirect an in-flight file. Interrupted relocations can be retried explicitly.

Alerts

?
An alert remains latched for its cycle until the tab receives trusted user input, or it stays visible for the configured timeout. Dismissing the alert does not stop monitoring. The monitor re-arms only after the condition becomes false, and the next MATCHED cycle triggers alerts and target handling again.
